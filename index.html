How the crypto works

  • Key pair (§1, §24): At election start, an RSA-2048 key pair is generated with crypto.subtle.generateKey. The public key is shared openly — any voter uses it to encrypt their vote. In production, the private key is never generated in or transmitted to a browser. In this demo it lives in localStorage — the intentional weak point.
  • Encryption (§10): Each vote is encrypted with RSA-OAEP (SHA-256) under the public key. Without the private key, the ciphertext is computationally infeasible to reverse. There is no IV or symmetric layer — RSA-OAEP provides its own randomisation internally.
  • Hashing (§12): Each ledger entry's hash is SHA-256(id + encryptedVote + timestamp), computed with crypto.subtle.digest. Anyone can recompute it to detect tampering.
  • Temporary Election Identity (§1–3): Each citizen has a long-term device key. For each election, a temporary election identity (EID) is derived locally: EID = "EID_" + SHA-256(citizenKey ∥ electionId)[0:20].toUpperCase(). The EID is the ballot ID visible on the public ledger. It is unlinkable across elections (different electionId each time), and does not reveal the citizen's real identity — only the fact that a ballot was cast.
  • Simulated ZKP (§10–11): Each ballot carries a Zero-Knowledge Proof field. In production: a real ZKP circuit (e.g. a Σ-protocol or Bulletproof) proves exactly one valid choice is encoded, publicly verifiable without revealing the vote. In this demo: a SHA-256 commitment sha256(ballotId + candidateIndex + nonce) binds the ballot identity to the specific candidate choice — proving exactly one valid selection was encoded (§12). For user-submitted ballots, ballotId = EID; for preloaded demo ballots, ballotId = PRE_0 etc.
  • Homomorphic tally (§23): In production, votes are summed in ciphertext space using homomorphic properties — no individual ballot is ever decrypted during tallying. Only the aggregate total is decrypted at the end. This demo simulates the outcome; individual decryption happens internally for display only.
  • Threshold decryption (§24): Final aggregate decryption uses threshold cryptography with a strict 7-of-10 quorum of independent key holders. No single holder possesses the full key. No full private key is ever reconstructed. This configuration assumes strong logistical coordination, mandatory participation protocols, and operational redundancy sufficient to avoid institutional deadlock. This demo simulates the ceremony; internally a single RSA key is used.

Demo limitations — what this page cannot replicate

Private key in localStorage — the only intentional vulnerability:
Open DevTools (F12) → Application → Local Storage → find electionPrivateKey.
Copy it into the console and run:
  const pk = await crypto.subtle.importKey("jwk", <paste JWK>, {name:"RSA-OAEP", hash:"SHA-256"}, false, ["decrypt"]);
  const plain = await crypto.subtle.decrypt({name:"RSA-OAEP"}, pk, Uint8Array.from(atob(<encrypted>), c=>c.charCodeAt(0)));
  new TextDecoder().decode(plain); // reveals the vote

This works only because there is no server. The private key had nowhere else to go. The public key being visible is not a vulnerability — RSA public keys are always public.
  • ZKP is a hash commitment, not a real ZKP circuit. It is not zero-knowledge in the cryptographic sense. The commitment binds the candidate index to the election identity — the hash itself does not reveal the vote, but the index is an input to the hash (simulated proof of validity).
  • Threshold is a UI simulation. Internally a single RSA key handles all decryption.
  • Homomorphic tally is simulated. The demo decrypts individually.
  • Device security is simulated. Production requires hardware-backed secure storage (HSM or Secure Enclave). The verification right (revealConsumed flag) lives in plain localStorage and is not hardware-bound (§28).
  • Terminal is a browser panel. A real reveal unit proves identity via official certificate, measured boot attestation, and a per-session challenge verified by backend infrastructure (§17). Backend verification of terminal integrity is not present in this demo.
  • Consumption uses a client-side localStorage flag in this demo. The session-persistence mechanism (write before display, clear after finalize) approximates interruption-resistant consumption handling for demo purposes, but the flag itself is not hardware-bound or server-verified. In production, the sole authority for recording consumption is the trusted reveal infrastructure (§19).

Voting right vs verification right (§9, §13, §19)

Two distinct rights exist per ballot:

  • Voting right (§9): consumed at block finalization — when the ballot is included in the sealed block. After this, the election identity cannot submit another ballot.
  • Verification right (§13): created at voting time, device-bound, single-use, non-exportable, non-recoverable. Consumed only when the ballot display cycle completes at the certified verification terminal. If the display does not complete, the right remains unused (§19). Consumed rights cannot be recovered.

Coercion resistance model (§22)

The system reduces vote-buying and coercion risk by preventing durable, transferable proof of how a voter voted:

  • No ballot reveal on the voter's personal device — ever.
  • Verification occurs only in a certified controlled environment: single-person access, no recording devices, controlled booth or equivalent (§15).
  • Brief display duration with automatic clearing — the ballot is visible for a constrained window only.
  • Single-use verification right — no replay, no second viewing.
  • No exportable proof path — nothing leaves the reveal chamber.

Goal: a voter can verify their own ballot but cannot reliably prove to a third party what they voted.

Verification infrastructure (§16–21)

  • Reveal units are stateless between sessions — no persistent ballot or session data is stored. All local reveal state is destroyed after each session (§16, §21).
  • Units are fully interchangeable. No single unit is critical infrastructure (§16).
  • The reveal model is semi-online: authorization is online, reveal runs locally, completion is synchronized afterward (§20). No live plaintext ballot transfer occurs during the display cycle.
  • Consumption is recorded only when the ballot display completes and the reveal monitor shuts down (§19). In production, the reveal infrastructure is the sole authority — the client never declares consumption. This demo approximates interruption-resistant consumption handling for demo purposes: the session is persisted before display begins and cleared only after finalization, so a page refresh cannot bypass consumption.
  • The consumption registry is ephemeral — active only during the 1-year verification window, then decommissioned (§19, §30).

Device security and lifecycle (§28–30, §35–37)

  • Hardware-backed storage (§28): in production, verification rights are stored in hardware-backed secure storage (HSM / Secure Enclave) and are non-exportable.
  • Lost device (§28): a lost device permanently destroys any unused current-election verification rights bound to it. There is no recovery mechanism.
  • Device migration (§36): migration through an official process immediately invalidates the old device. Current-election verification rights do not migrate and are permanently lost.
  • Re-enrollment (§35): official re-enrollment permanently disables the previous device for all future system use.
  • Election archival (§30): one year after election closure, verification is permanently disabled. The election ledger is archived. The consumption registry is decommissioned. No further ballot verification is possible.

Physical vs digital separation (§26–27)

Digital and physical voting are strictly separated channels. A citizen may use only one. Entering official digital enrollment immediately and irreversibly disables the physical voting channel for that citizen. This commitment cannot be undone.

Audit logging compliance (§31)

The system event log records only: terminal or reveal unit identifier, timestamp, and session outcome (authorized, consumed, cleared). The log must never contain citizen identity, vote content, readable ballot linkage, or data sufficient to reconstruct who verified what. This demo's system log complies — no voter name, candidate choice, or EID-to-vote mapping is ever written to the log.

Verification steps — open standard (§25)

  1. Confirm each ballot's election identity (EID) appears in the sealed block with status FINAL.
  2. Inspect the simulated ZKP field as a demo placeholder for a real publicly verifiable ballot-validity proof. In this prototype, the ZKP field is a private hash commitment rather than a real zero-knowledge proof (§11).
  3. Recompute SHA-256(id + encryptedVote + timestamp) for each entry and confirm it matches the published hash.
  4. After the threshold decryption ceremony (7/10 key holders), sum the decrypted vote totals independently to reproduce the tally.
  5. No backend trust is required at any step — all inputs are publicly available on the ledger (§25).
