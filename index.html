Demo notice: This prototype uses real RSA-2048-OAEP asymmetric encryption and real SHA-256 hashes via the browser's Web Crypto API. Votes are encrypted under the election's public key — anyone can encrypt, but only the holder of the private key can decrypt. In this demo the private key is stored in localStorage (the intentional weak point). In a real deployment it would never leave a server. See the Security Transparency section below.

How the crypto works

  • Key pair: At election start, an RSA-2048 key pair is generated with crypto.subtle.generateKey. The public key is shared openly — any voter uses it to encrypt their vote. The private key is held by the election authority and used only at close to decrypt and tally.
  • Encryption: Each vote is encrypted with RSA-OAEP (SHA-256) under the public key. Without the private key, the ciphertext is computationally infeasible to reverse. There is no IV or symmetric layer — RSA-OAEP provides its own randomisation internally.
  • Hashing: Each ledger entry's hash is SHA-256(id + encryptedVote + timestamp), computed with crypto.subtle.digest. Anyone can recompute it to detect tampering.

The demo limitation — where the private key lives

How to "break" this demo:
1. Open DevTools (F12) → Application → Local Storage → this page
2. Find the key named electionPrivateKey — that is the RSA private key as a JWK.
3. Copy it into the console and run:
  const pk = await crypto.subtle.importKey("jwk", <paste JWK>, {name:"RSA-OAEP", hash:"SHA-256"}, false, ["decrypt"]);
  const plain = await crypto.subtle.decrypt({name:"RSA-OAEP"}, pk, Uint8Array.from(atob(<encrypted>), c=>c.charCodeAt(0)));
  new TextDecoder().decode(plain); // reveals the vote

This works only because there is no server. The private key had nowhere else to go. The public key being visible is not a vulnerability — RSA public keys are always public.
What a real system would do differently:
In a production deployment the RSA key pair would be generated server-side. The public key would be published openly (anyone can encrypt). The private key would never be transmitted to the browser — it lives only in the server's memory or an HSM. After the election closes, the authority decrypts server-side and publishes the plaintext votes alongside a cryptographic proof, so any third party can independently re-tally. Inspecting localStorage, the network tab, or any client-side storage would reveal only ciphertext and the public key — neither of which helps an attacker.

Verification steps (open standard)

  1. Record the ledger entry: (id, encryptedVote, hash)
  2. Recompute hash = SHA-256(id + encryptedVote + timestamp) and confirm it matches the published value.
  3. After the election closes and the authority publishes the private key: decrypt each entry and confirm the plaintext vote.
  4. Sum all decrypted votes independently to reproduce the tally without trusting the authority's count.
